# Руководство для тренера

## Цель
Раздать участникам понятный сценарий, в котором они быстро запускают окружение, проводят кейсы и фиксируют выводы.

## Быстрый старт (5 минут)
1. Клонируем и запускаем:
   ```bash
git clone <repo>
cd workshop_devops_with_app
docker compose up -d --build
   ```
2. Проверяем сервисы:
   ```bash
docker compose ps
curl http://localhost:5000/health
open http://localhost:9090
open http://localhost:3000
   ```
3. В Grafana логинимся `admin/workshop` и проверяем datasource `Prometheus`.

## Структура мастер-класса (1,5 часа)
1. Введение (5 мин)
   - Что такое reliability/observability
   - Архитектура (Flask + PostgreSQL + Prometheus/Grafana)
   - Цели: быстрее обнаруживать, реагировать и восстанавливаться.
2. Запуск и проверка (10 мин)
   - Запустить `docker compose up -d --build`
   - Проверить `health`, Prometheus, Grafana.
3. Работа с кейсами (60 мин)
   - Кейс 1: latency
   - Кейс 2: error rate
   - Кейс 3: медленная БД
   - Кейс 4: memory leak
   - Кейс 5: chaos mix
4. Итоги и Q&A (15 мин)
   - Какие гипотезы подтвердились?
   - Что улучшить в архитектуре?

## Стандартный цикл для каждого кейса
1. Baseline: измеряем текущие метрики (p95, error_rate, использование CPU/memory).
2. Inject: инъектим отказ через API.
3. Detect: смотрим Grafana/Prometheus, выявляем отклонения.
4. Respond: формируем действия (сброс состояния, масштабирование, retries, circuit breaker).
5. Recover: сбрасываем `/api/reset`, возвращаем baseline.

## Команды для кейсов
### 1. Latency
```bash
bash scripts/run_latency.sh
```
Открыть Grafana и проверить p95/p99, RPS.

> Почему именно p95/p99/RPS?
> - p95/p99 показывают реальный опыт пользователей при редких, но ощутимых задержках.
> - RPS показывает, можем ли мы сохранить throughput при деградации и не упасть в провал нагрузки.
> - Вместе: latency + RPS дает контекст, что приведено к проблеме (деградация vs блокировка).

### 2. Error rate
```bash
bash scripts/run_error_rate.sh
```
Проверить `error_count`, `error_rate`.

> Почему error rate?
> - Это прямой показатель того, что система деградировала для пользователей.
> - Позволяет понять, действует ли fail-fast/обратный откат при отказы.
> - С помощью графика error rate можно определить, когда стоит переключаться на fallback.

### 3. Медленная БД
```bash
bash scripts/run_db_slow.sh
```
Посмотреть `fintech_active_transactions`, задержки БД.

Когда active_transactions растут, это значит, что запросы не успевают завершаться (блокировки, медленные запросы, исчерпан пул соединений).
В отличие от p95 latency, это не только про время ответа, а про количество “висящих” запросов в системе.
Если active_transactions растут, а API всё ещё отвечает, это именно “проблема БД/пула”, а не только “плохой код endpoint’a”.
Для SRE это KPI: сколько запросов в очереди? Если он растёт, значит нагрузку не разгребают и скоро начнётся каскад (timeouts, 5xx, OOM).

### 4. Утечка памяти
```bash
bash scripts/run_memory_leak.sh
```
Проверить `container_memory_usage_bytes`, `restarts`.

> Почему memory usage + restarts?
> - Утечка памяти — В результате кумулятивно используется всё больше RAM, и через какое‑то время контейнер достигает лимита и система убивает процесс (OOMKilled).

  Это медленный инцидент: память постепенно растёт, и критическое состояние проявляется не мгновенно. Поэтому важно смотреть не только точечное значение, но и тренд, чтобы оценить, через сколько времени потребуется вмешательство.

  Рестарты контейнера сигнализируют, что приложение не справляется, и система должна корректно отработать тревоги (alerts) и процессы релизов (rollout), чтобы восстановить стабильность.

### 5. Chaos Mix
```bash
bash scripts/run_chaos_mix.sh
```
Сравнить последствия.

Chaos mix проверяет устойчивость всей системы, когда проблемы идут сразу в нескольких слоях (API, БД, память).
Он помогает отличать локальные отказы (частичный сбой одного компонента) от каскадных (когда один сбой тянет за собой другую часть системы).
И главный урок: теперь можно оценить, какие защитные меры реально помогли (retry/circuit-breaker/трассировка), а какие — нет.

## Проверка состояния железа (ресурсов)
- Важно знать, что в воркшопе мы смотрим в основном метрики контейнеров и приложения через Prometheus/cAdvisor.
- Для cAdvisor: `http://localhost:8080/` и метрики `container_cpu_usage_seconds_total`, `container_memory_usage_bytes`.
- Если нужно смотреть хост-систему (за пределами контейнера) — используйте `top`, `htop`, `vmstat` на хосте.
- Это выходит за рамки базового мастер-класса, но помогает подтвердить, что проблема не в железе (CPU/IO) 

## Ведущему по ходу мастер-класса
- Всегда отbaseline: именно baseline позволяет понять, что поменяется.
- Наблюдать за метриками в реальном времени (Grafana).
- Обсуждать, почему сервисы падают/перезапускаются.
- Заставить команду сформулировать runbook (что делать, если p95 > 500ms).

## Итоговый чек-лист для команд
- [ ] Сервис доступен (`/health`)
- [ ] В случае кейса метрики явно изменились
- [ ] Алерты/метрики объясняют причину
- [ ] Выполнен `/api/reset` после каждого кейса
- [ ] Написан runbook (3–5 шагов)
