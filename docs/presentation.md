# Fintech DevOps Workshop

## 1. Название
**Fintech Reliability Sprint: Chaos, Monitoring & Recovery**

## 2. На кого рассчитано
- QA Engineers, DevOps, SRE, backend-разработчики
- Команды, которые хотят быстро оценить отказоустойчивость и SLO
- Участники c базовым Docker и HTTP

## 3. Кто ведёт
**Имя тренера**: [Ваше имя]
- Опыт: построение наблюдаемости в финтех-проектах
- Фокус: видеть инциденты раньше, восстанавливаться быстрее

## 4. План мероприятия (90 минут)
1. Введение: цели, архитектура, метрики (10 м)
2. Запуск окружения и проверка (10 м)
3. Кейс 1: Latency (15 м)
4. Кейс 2: Error Rate (15 м)
5. Кейс 3: DB Slow (15 м)
6. Кейс 4: Memory Leak (15 м)
7. Кейс 5: Chaos Mix (10 м)
8. Итоги + Q&A (10 м)

## 5. Архитектура приложения
```
[User] --> [Fintech API (Flask)] --> [PostgreSQL]
                  |                     
                  |-> [Prometheus scrape /metrics]
                  |-> [Grafana dashboards]
                  |-> [cAdvisor (container metrics)]
```

### Компоненты
- **App** — Flask + injection API (`/api/failures`, `/api/reset`, `/metrics`)
- **DB** — PostgreSQL
- **Prometheus** — сбор метрик API + контейнеров
- **Grafana** — визуализация + дашборды
- **cadvisor** — контейнерные метрики

## 6. Краткое описание кейсов
- **Кейс 1: Рост latency** — инъекция задержки, проверка p95/p99, реакция ✅
- **Кейс 2: Рост error rate** — случайные 5xx, анализ влияния на доступность ✅
- **Кейс 3: Медленная БД** — активные транзакции, дедлок, нагрузка ✅
- **Кейс 4: Утечка памяти** — OOM, рестарты контейнера, ресурсное наблюдение ✅
- **Кейс 5: Chaos Mix** — несколько проблем вместе, приоритизация, recovery ✅

## 7. Что участник унесёт
- Практический workflow: baseline → inject → detect → respond → recover
- Набросок runbook для быстрого восстановления
- Чёткая связь метрик, алертов и инцидента

---

> Pro tip: после каждого кейса записываем 3 вещи: что изменилось, как обнаружили, что сделали.
