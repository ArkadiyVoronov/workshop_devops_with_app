# Fintech DevOps Workshop

## 1. Название
**Fintech Reliability Sprint: Chaos, Monitoring & Recovery**

## 2. На кого рассчитано
- QA Engineers, DevOps, SRE, backend-разработчики
- Команды, которые хотят быстро оценить отказоустойчивость и SLO
- Участники c базовым Docker и HTTP

## 3. Кто ведёт
**Имя тренера**: [Ваше имя]
- Опыт: построение наблюдаемости в финтех-проектах
- Фокус: видеть инциденты раньше, восстанавливаться быстрее

## 4. План мероприятия (90 минут)
1. Введение: цели, архитектура, метрики (10 м)
2. Запуск окружения и проверка (10 м)
3. Кейс 1: Latency (15 м)
4. Кейс 2: Error Rate (15 м)
5. Кейс 3: DB Slow (15 м)
6. Кейс 4: Memory Leak (15 м)
7. Кейс 5: Chaos Mix (10 м)
8. Итоги + Q&A (10 м)

## 5. Архитектура приложения
```
[User] --> [Fintech API (Flask)] --> [PostgreSQL]
                  |                     
                  |-> [Prometheus scrape /metrics]
                  |-> [Grafana dashboards]
                  |-> [cAdvisor (container metrics)]
```

### Компоненты
- **App** — Flask + injection API (`/api/failures`, `/api/reset`, `/metrics`)
- **DB** — PostgreSQL
- **Prometheus** — сбор метрик API + контейнеров
- **Grafana** — визуализация + дашборды
- **cadvisor** — контейнерные метрики

## 6. Краткое описание кейсов
- **Кейс 1: Рост latency**
  - Запуск: `bash scripts/run_latency.sh`.
  - Как inject: скрипт вызывает `/api/failures?latency=200` (или аналог), добавляя искусственную задержку.
  - Что inject: задержка в endpoint, p95/p99 растут.
  - Что видим на графиках: скорость ответа (p95/p99) поднимается, RPS может упасть, alert `HighLatency`.
  - Почему: задержка влияет на пользовательский опыт и SLO.
- **Кейс 2: Рост error rate**
  - Запуск: `bash scripts/run_error_rate.sh`.
  - Как inject: скрипт вызывает `/api/failures?error_rate=0.4`, генерируя ~40% 5xx.
  - Что inject: искусственные 5xx/ошибки.
  - Что видим на графиках: `fintech_error_count` и `error_rate` вверх, проценты 5xx поднимаются, alert `HighErrorRate`.
  - Почему: ошибки напрямую снижают доступность и доверие пользователей.
- **Кейс 3: Медленная БД**
  - Запуск: `bash scripts/run_db_slow.sh`.
  - Как inject: скрипт щёлкает `/api/failures?db_delay=200` и/или добавляет нагрузку на транзакции.
  - Что inject: задержка/блокировки в БД.
  - Что видим на графиках: `fintech_active_transactions` растёт, latency поднимается, связка с DB latency; alert `DBSlowTransactions`.
  - Почему: медленная БД приводит к накоплению транзакций и каскаду таймаутов.
- **Кейс 4: Утечка памяти**
  - Запуск: `bash scripts/run_memory_leak.sh`.
  - Как inject: скрипт вызывает `/api/failures?memory_leak=1`, приложение начинает удерживать объекты.
  - Что inject: постепенное увеличение использования памяти.
  - Что видим на графиках: `container_memory_usage_bytes` тренд вверх, `container_restart_count` может расти; alert `MemoryPressure`.
  - Почему: память — ресурс, исчерпаемый со временем; OOM/перезапуск резко влияет на SLA.
- **Кейс 5: Chaos Mix**
  - Запуск: `bash scripts/run_chaos_mix.sh`.
  - Как inject: скрипт комбинирует несколько параметров (`latency`, `error_rate`, `db_delay`, `memory_leak`) в API.
  - Что inject: несколько аномалий одновременно (latency, ошибки, DB нагрузка, память).
  - Что видим на графиках: несколько дашбордов одновременно ухудшаются; сравниваем, что сработало (алерты + recovery).
  - Почему: проверка устойчивости и приоритетов в реальном multi-failure сценарии.

## 7. Что участник унесёт
- Практический workflow: baseline → inject → detect → respond → recover
- Набросок runbook для быстрого восстановления
- Чёткая связь метрик, алертов и инцидента

## 8. Сигналы и реакции (быстрая табличка)
| Сигнал | Что смотреть | Почему важно | Первое действие |
|---|---|---|---|
| p95/p99 растёт | latency buckets, RPS | может быть деградация UX и SLO | проверить код/зависимости, откатить/увеличить ресурсы |
| error_rate растёт | error_count, error_rate | падает доступность, пользователи получают 5xx | найти источник ошибки, временно выключить feature, зайти в логи |
| active_transactions растёт | fintech_active_transactions, DB latency | очередь запросов растёт, риск таймаутов | уменьшить нагрузку, проанализировать медленные запросы, увеличить пул |
| memory usage растёт | container_memory_usage_bytes, restart_count | OOM/рестарты → потеря доступности | перезапустить, выставить лимиты, искать утечку |
